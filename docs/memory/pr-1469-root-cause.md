ipython/ipykernel **#1469** ("Enable ProactorEventLoop on Windows", by NewUserHa; sibling of the team's own [[upstream-pr-filed]] #1529) — root-caused locally 2026-06-18 in worktree `M:\claud_projects\ipykernel-1469` (branch `pr-1469`, head `4032796`). `test_debugger.py::test_stop_on_breakpoint` is the failing test.

**Proven (primary evidence, debugpy's own logs via `DEBUGPY_LOG_DIR`):** the bug is **ipykernel's read side, NOT debugpy**. On the failing run debugpy hits the breakpoint, suspends MainThread, and the adapter writes `Client[1] <-- {event:"stopped"}` @649ms — but ipykernel's **idle control-thread `ProactorEventLoop` never wakes to read it** off the (still-open) zmq socket; it waits the full 100s. Earlier debug frames (seq 1–17: attach/setBreakpoints/threads…) were all read fine because the loop was actively `await`-driven then; `stopped` is the first frame arriving while the control loop is idle.

**2×2 (same instrumented build, only the named variable changes):** Proactor+3.13 → FAIL; Proactor+3.11 → PASS; forced-Selector+3.13 → PASS. ⇒ failure = **intersection Proactor × Python ≥3.12**, matching CI (Win 3.10/3.11 pass, 3.12/3.13/3.14/pypy fail).

**MECHANISM — PROVEN 2026-06-18 (faulthandler + 3.11 mirror):** asyncio `ProactorEventLoop` has **no** native `add_reader` (both versions `NotImplementedError`) → tornado wraps it with `AddThreadSelectorEventLoop`, whose **"Tornado selector"** helper thread does `select()` then `call_soon_threadsafe` (asyncio.py `_run_select:625`) to wake the loop. ipykernel exempts its Control/IOPub threads from debugpy but NOT tornado's helper. **3.13 faulthandler stack of the wedged kernel = the helper frozen at `_run_select:625` inside `call_soon_threadsafe` → `do_wait_suspend`** (select already returned; frozen before it can wake the loop; the Proactor loop sits in IOCP `_poll` forever = deadlock). **3.11 = helper completes call_soon_threadsafe, `Control HANDLE_SELECT` runs** → stopped read → pass. **Version delta = pydevd backend:** 3.11 `sys.settrace` (per-thread) doesn't freeze the tornado-owned helper; 3.13 `sys.monitoring` (PEP669, interpreter-global; compiled `_pydevd_sys_monitoring_cython` loaded) instruments all threads incl. the helper. (`Marking thread suspended: Tornado selector` appears on BOTH — wrong test; "frozen AT call_soon_threadsafe" only on 3.13 — right test.)

**FIX A = BUILT & TESTED 2026-06-18 (the ship).** Run ALL ipykernel *service* loops on a `SelectorEventLoop` (native add_reader, no "Tornado selector" helper) and keep ONLY the main/user-code loop on Proactor (so #1468 `create_subprocess_exec` works). = pre-#1469 loop type for every loop except main. **2 tracked files, ~20 lines, win32-guarded, NO pydevd coupling:** `thread.py` new `make_selector_io_loop()` (`IOLoop(make_current=False, asyncio_loop=asyncio.SelectorEventLoop())`) used by `BaseThread` (Control/Shell-channel/**subshells** — `SubshellThread(BaseThread)`); `iostream.py` `IOPubThread` uses it (IOPub MUST be Selector — control republishes `stopped` onto IOPub's zmq event-pipe `add_reader`, same freezable helper otherwise). Main loop (`IOLoop.current()`) untouched.
